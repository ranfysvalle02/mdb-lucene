the macro-strategy — in one sentence

It isn't fastest vector retrieval.
It isn't even the database that added vectors to its query language.
It's that MongoDB Atlas absorbs complexity into the platform
and builds convenience into the query language — and has been doing it, one capability at a time, since 2009. That's the strategy. Vector search is one expression of it. So is everything else the platform has absorbed in the last fifteen years — geospatial, graph traversal, ACID across shards, change streams, time-series, full-text search, queryable encryption, stream processing, hybrid retrieval, in-database embeddings + reranking. Every wave the industry framed as "you'll need a separate database for this" got the same answer: extend the query layer. Never rebuild the foundation. Make the database you already have smarter. That's why /beyond-the-vector-database is the only contest left — the engine inside $vectorSearch is commodity (panel 2 below proves it), and the only thing left to differentiate on is the strategy that brought vector search into the same place as everything else.

Lucene proves the vector engine is commodity → therefore → the only game left is what the database absorbs beyond the engine.

Premise on the left = panel 2 below (engine equivalence, four decimal places). Conclusion on the right = panel 1 below (the composable pipeline). The arrow in the middle is the entire post.

An analogy. The competing data layer has been bolting JSONB columns, "relational duality views," vector indexes, and stream connectors onto a relational engine designed in the 1970s. Engineering marvels, all of them. But putting an electric motor in a 1980s station wagon doesn't make it a Tesla. The document model wasn't a feature retrofitted onto a relational engine — it was the engine, and every absorption since 2009 was an extension of the same engine, not a graft onto a different one. That's why the strategy compounds. (Borrowed, with admiration, from The Shape of Memory · Chapter I.)

how to read this page

"This page treats the developer not as a consumer to be sold to, but as an architect trying to prevent their AI infrastructure from collapsing into a pile of duct-taped microservices."

What follows isn't a vendor scorecard. It's a runnable laptop demo, a timeline of fifteen years of absorptions, a platform map with sixteen tiles you can click, and an honest appendix about when a specialized vector database actually is the right pick. The work it asks you to do is the work you were already doing in your head when you opened your laptop today: sketching the data layer for a system you don't want to be paged about.

OK but what does any of this actually mean? Let's get concrete. Picture an actual AI agent. A user types "change my Tuesday flight to Thursday." Behind the scenes, five things have to happen, in roughly this order:

  1. Look up your booking. — a normal database query against the airline's records ("find the booking for user X on Tuesday")
  2. Read what's been said in this chat so far. — the agent didn't start fresh; it has the conversation up to now ("you said Tuesday, you want Thursday, you mentioned a window seat")
  3. Look up the airline's reschedule policy.vector search — phrased semantically: "rescheduling rules" should match a doc titled "Change Fee Schedule." This is the step the term "vector database" was named for.
  4. Recall what you usually prefer. — you always pick aisle seats, you hate red-eyes, you're flexible on dates but not on departure airports
  5. Check what worked last time. — last time the agent rescheduled a flight, did the airline's API actually accept the request, or did it 500?

A "vector database" handles step 3. That's it. One out of five. The other four steps live in completely different systems — a Postgres for step 1, a Redis for step 2, a DynamoDB for step 4, a ClickHouse for step 5 — or they live in your application code held together with HTTP calls and prayers, plus a CDC pipeline keeping all of them in sync.

An "agentic database" is shorthand for "the document model + the aggregation framework — the same one that absorbed geo, graph, time-series, full-text, vector, stream processing, and encrypted-yet-searchable BSON, one capability at a time — applied to all five of those state types at once." That's MongoDB Atlas. And the five-types story is the smallest version of the real story. The real story is the strategy that brought all five types into the same framework: absorb complexity into the platform, build convenience into the query language, never rebuild the foundation. (The full version of that strategy, on a timeline, is in the unbroken-thread panel directly below.)

A vector database is to an agentic database what an engine is to a car. The engine is critical. It's where the horsepower lives. It's expensive. It also doesn't have wheels, seats, brakes, AC, headlights, or a steering wheel. You can't drive an engine. You drive the car. When you ship an AI agent, you're shipping a car. The vector store is the engine.

show me the MQL — the same five steps, then the punchline 5 queries → 1 pipeline

Each step on its own first — they look like the boring everyday queries any application developer would write. Then watch what happens at the bottom when the same five steps become five stages of one aggregation pipeline.

1"Look up your booking." — operational record · findOne

db.bookings.findOne({
  user_id:  "u_8421",
  depart_at: { $gte: ISODate("2026-05-19"),
               $lt:  ISODate("2026-05-20") }
})
// → { _id: "BK-7724", flight_no: "AA1132", seat: "27A",
//     fare_class: "Q", origin: "JFK", destination: "SFO" }

2"Read what's been said in this chat so far." — session state · scoped find + sort

db.chats.find(
  { session_id: "sess_3f9c", user_id: "u_8421" }
).sort({ ts: 1 })
// → [
//     { role: "user",  text: "change my Tuesday flight to Thursday" },
//     { role: "agent", text: "Got it — same time of day OK?" },
//     { role: "user",  text: "yes, and window if you can" }
//   ]

3"Look up the airline's reschedule policy." — vector recall · $vectorSearch with pre-filter pushdown

db.policies.aggregate([
  { $vectorSearch: {
      index:       "policy_index",
      path:        "embedding",
      queryVector: embed("rescheduling rules"),
      filter:      { airline: "AA" },   // pushed INTO HNSW, not after
      limit:       3
  }},
  { $project: { title: 1, body: 1,
                _score: { $meta: "vectorSearchScore" } }}
])
// → [
//     { title: "Change Fee Schedule",          _score: 0.84 },
//     { title: "Same-Day Confirmation Policy", _score: 0.71 }
//   ]

4"Recall what you usually prefer." — long-term memory · findOne on the user's preferences

db.user_memory.findOne(
  { user_id: "u_8421" },
  { preferences: 1 }
)
// → { preferences: {
//       seat:           "aisle",
//       avoid_red_eyes: true,
//       date_flexible:  true,
//       airports_strict: ["JFK", "LGA"]   // never EWR
//   }}

5"Check what worked last time." — tool traces · time-series collection · most-recent successful call

db.tool_traces.find({
  user_id: "u_8421",
  tool:    "airline_reschedule_api",
  ts:      { $gte: ISODate("2026-04-01") }
}).sort({ ts: -1 }).limit(1)
// → { ts: ISODate("2026-04-12T14:22Z"),
//     tool: "airline_reschedule_api",
//     status: 200, latency_ms: 1840,
//     response: { confirmation: "AA-2104" } }
the punchline — the same five steps, composed into one pipeline, in one round trip 
// One agent turn. One aggregation pipeline. One round trip.
db.bookings.aggregate([

  // Step 1 — operational record (the seed for everything else)
  { $match: {
      user_id:  "u_8421",
      depart_at: { $gte: ISODate("2026-05-19"),
                   $lt:  ISODate("2026-05-20") }
  }},

  // Step 4 — long-term memory, joined onto the user
  { $lookup: {
      from: "user_memory", localField: "user_id",
      foreignField: "user_id", as: "memory"
  }},

  // Step 2 — session/chat state, ordered, scoped to this session
  { $lookup: {
      from: "chats",
      let:  { uid: "$user_id" },
      pipeline: [
        { $match: { $expr: { $and: [
            { $eq: ["$user_id",    "$$uid"] },
            { $eq: ["$session_id", "sess_3f9c"] }
        ]}}},
        { $sort: { ts: 1 } }
      ],
      as: "chat"
  }},

  // Step 3 — vector recall over the policy corpus, pre-filter pushed INTO HNSW
  { $lookup: {
      from: "policies",
      pipeline: [
        { $vectorSearch: {
            index:       "policy_index",
            path:        "embedding",
            queryVector: embed("rescheduling rules"),
            filter:      { airline: "AA" },
            limit:       3
        }},
        { $project: { title: 1, body: 1 } }
      ],
      as: "policy"
  }},

  // Step 5 — most recent successful call to this tool for this user (time-series)
  { $lookup: {
      from: "tool_traces",
      let:  { uid: "$user_id" },
      pipeline: [
        { $match: { $expr: { $eq: ["$user_id", "$$uid"] },
                    tool: "airline_reschedule_api",
                    ts:   { $gte: ISODate("2026-04-01") } }},
        { $sort:  { ts: -1 } },
        { $limit: 1 }
      ],
      as: "last_attempt"
  }},

  // Final shape — exactly what the agent's reasoning step needs as input.
  { $project: {
      booking:      { _id: "$_id", flight_no: "$flight_no",
                      seat: "$seat", origin: "$origin",
                      destination: "$destination" },
      memory:       { $first: "$memory" },
      chat:         1,
      policy:       1,
      last_attempt: { $first: "$last_attempt" }
  }}

])
// → ONE document containing operational + memory + chat + policy + traces.
//   ONE round trip. ONE query language. ONE ops surface. ONE auth model.
//   ONE backup. ONE SOC2 audit. ONE on-call rotation.

That's it. $match + four $lookups + a $project — operators that have existed in MongoDB since long before "AI agent" was a job title. The $vectorSearch stage is one of the four lookups. It sits between the chat-history lookup and the tool-trace lookup, with the same syntax the other lookups use, against the same database. That is what "vectors are chapter 6, not chapter 1" looks like in code. Vector recall is a stage. The aggregation framework already knew how to do four other kinds of stages; it now also knows how to do this one; it composes them all without you needing to write any glue.

Now picture the same five steps in a split-store architecture: a relational call, a cache call, a pure-vector recall call, another operational call, an observability call — five surfaces, five auth models, five SLAs, five places drift can show up, plus a sync path keeping semantic recall aligned with the booking row every time the airline's policy team updates a doc. The agentic-database argument is the difference between those two pictures. It's not "whose raw top-K wins a micro-benchmark." It's "the second picture is one query and the first picture is a project."

The composable-pipeline panel further down on this page is the same shape against simpler data (movies + reviews) that you can run on your laptop in real time, with a "raw Lucene + glue" column next to it that quantifies the cost of the alternative. Every line of MQL above is a stage in the same aggregation framework that absorbed geo, graph, time-series, full-text, vector, stream processing, and queryable encryption — one capability at a time. This block is the macro-strategy expressed in twenty-five lines of query.

Below is the formal version of "the five things" the walkthrough just illustrated. Same content, slightly more technical wording — so you can map them to the systems you already run.

01 · operational data

"Look up your booking" — boring rows-and-columns

Orders, users, products, tickets, inventory. The OLTP records the agent has to be consistent with — and write back to in the same transaction.

02 · vector recall

"Look up the policy" — semantic search

The thing "vector database" was named for. $vectorSearch on a field of the document. Filter pushed into HNSW (pre-filter, not post-filter). Composes with $lookup, $rankFusion, the rest of the pipeline.

03 · session state

"Read this chat so far" — short-term memory

Conversation history, current task, in-progress tool plans. Constantly read, constantly appended to, filtered by session_id. Often also embedded for "did we already talk about this?"

04 · long-term memory

"What you usually prefer" — the agent's notes on you

User preferences, recurring topics, prior decisions. Hybrid of structured fields and free-text + embeddings, scoped per-tenant via filter pushdown.

05 · tool traces

"What worked last time" — observability + replay

Every tool call the agent made, with args and result. Time-series. Queryable by run-id or by semantic similarity. Just another collection — with its own embedding column.

One last thing before the demo: the retrieval engine itself is portable — Atlas $vectorSearch is Lucene HNSW under the hood — the same library family that powers most Lucene-backed search stacks — and agrees to within noise with other mature HNSW implementations for the same vectors. The engine-equivalence panel further down proves it to four decimals. So the "fastest top-K" benchmark people obsess over is comparing engines that agree on the answer. The contest worth having is the one above — the five-step, one-database-or-many question. The composable-pipeline panel below is what step 3 looks like composed with steps 1, 4, and 5 in one query, vs. what it costs as separate round trips and glue.

the macro-strategy on a timeline  ·  click any node for the absorption story

Step back from "vector search" for a minute.
Vector search is one expression of a strategy MongoDB has been running, one capability at a time, since 2009.

The walkthrough above frames MongoDB as "a database that holds all five state types an agent needs." That's true — but it's a single frame from a much longer film. The longer film is the one The Shape of Memory narrates in five chapters: every "you'll need a separate database for that" wave the industry has named in the last fifteen years got the same answer from the same team — extend the query layer; never rebuild the foundation; make the database you already have smarter.

the strategy as a single recurring engineering decision

"This new data shape is just JSON.
Let's extend the aggregation framework to understand it natively."

That's it. That's the move. Sixteen years of database history compressed into one decision, made over and over again at the right time:

  • Coordinates are nested arrays  →  $geoNear
  • Edges are references  →  $graphLookup
  • Time-series readings are timestamped objects  →  native time-series collections
  • Inverted-index terms are arrays of strings  →  $search
  • Embeddings are arrays of floats  →  $vectorSearch
  • Encrypted fields are opaque BSON  →  queryable encryption
  • Stream events are documents in motion  →  Atlas Stream Processing

Same answer to the same question, sixteen years running. The timeline below is what that decision looks like as a calendar. Click any node for the absorption story.

Every node above was the same answer to the same question. "You'll need a spatial database for that."  →  $geoNear. "You'll need a graph database for that."  →  $graphLookup. "You'll need a time-series database for that."  →  native time-series collections. "You'll need Elasticsearch for that."  →  $search, with Lucene running inside the cluster, the index tailing the oplog. "You'll need a vector database for that."  →  $vectorSearch, on the same Lucene, on the same oplog tail, in the same aggregation framework. "You'll need Flink for that."  →  Atlas Stream Processing, the same pipeline language run continuously on data in motion. "You'll need a separate embedding API for that."  →  Voyage AI in the database, with autoEmbed as a field type. Same answer. Same framework. Same model. Sixteen years of compounding. That's the macro-strategy: absorb complexity into the platform; build convenience into the query language. Vector search is the latest expression of it. Whatever the next data shape turns out to be — whatever wave you'll be told you need a separate database for in 2027 — will land in the same place, the same way. Because every future data pattern still has to express nested structure, relationships, and evolving shape; and that's still native JSON.

"An agent's memory isn't a flat vector — it's summaries, entity graphs, temporal context, embeddings, and encrypted preferences, layered together. The document was always the natural shape for that kind of structured richness. It didn't require a new architecture. It required the one that was already there." — The Shape of Memory · Chapter VI · The Unbroken Thread

That paragraph is the closing line of the strategic essay. The demo on this page is what that line looks like rendered as a working pipeline. The "rescheduling your flight" walkthrough above isn't really about vector search. It's about a single aggregation pipeline reading from documents that happen to hold a vector column AND a session document AND an operational record AND a memory entry AND a tool-trace time-series — all of which are just JSON, all of which the aggregation framework already understands. The five state types compose in one query for the same reason geo + graph + time-series did a decade ago, and for the same reason whatever-comes-next will compose with them too: one foundation, one query language, one ops surface, one capability at a time. Vector recall is one column. The strategy that brought it there alongside everything else is the actual product.

what this 20-doc demo can show vs. what it can't

Built to make three things obvious: (1) the engine returns the same answer either way (engine equivalence — panel 2 proves it to four decimals), (2) one pipeline composes more cleanly than four glue steps, and (3) filter pushdown preserves recall on selective predicates (try the fantasy or crime · avg ≥ 4 chips).

Deliberately not modelled here: production CDC cost, auth/SOC2 surfaces, multi-tenant scoping at scale, or 10⁶+-vector latency. Those live in the bill-of-materials and the architectural matrix below — not in the millisecond counter. If raw Lucene comes out a millisecond ahead on this dataset, that's the demo working as intended. On 20 docs in localhost the engine cost is roundoff; what scales with N agent steps is the round trips, the query languages, and the sync pipelines you own.

        architecture · the five seams in an agent's data layer
        
      
        These aren't vendor traps — they're the places split stacks leak integration tax
        
      
        
        Each seam is a real engineering trade-off: a class of bug, an extra system, or a recurring cost that shows up
        after the POC. Pressure-test your architecture against them. Most split-store shapes answer yes to one seam,
        partially to a couple more, no to the rest. MongoDB Atlas is the shape that can answer yes
        across all five — not because it's a "better vector database," but because it's built as one composable data layer.
      
        

      
        
        
        
          Seam 1 · unified state
          Can vector recall, operational records, session state, long-term memory, and tool-call traces all live in one database?
          If no: a CDC pipeline you maintain forever, plus N−1 extra ops surfaces. Whose on-call rotation is that?
          ✓ Atlas: collections + time-series + change streams. All five live in one cluster.
        
        
        
        
          Seam 2 · filter in the walk
          Does the vector index push predicates into HNSW (pre-filter), not after the top-K returns (post-filter)?
          If no: on selective filters (per-tenant scoping, per-user ACLs), recall doesn't degrade — it collapses. Demo it on a 1%-selectivity predicate.
          ✓ Atlas: filter-type fields in the vectorSearch index. The Atlas column above proves it.
        
        
        
        
          Seam 3 · composition without an HTTP seam
          Can I express vector recall + filter + join + hybrid fusion + server-side rerank + aggregate + project in one query, in one round trip — with the entire query processor on the same ops surface as the data it composes?
          If no: every agent turn pays for the round-trip count in latency and cost; orchestration lives in your app forever; and hybrid fusion + the reranker become two more moving parts with two more SLAs. The query processor is not portable between stacks.
          ✓ Atlas: $vectorSearch + $rankFusion + $rerank + $lookup are stages of the same aggregation pipeline. The panel above shows it; the engine-eq panel below proves the kitchen is the same — only the front of house differs.
        
        
        
        
          Seam 4 · freshness as a property
          Does the search index stay fresh from the same writes that update the operational record — automatically, with no separate writer?
          If no: you own a CDC pipeline (resume tokens, dead-letter queues, replay tooling, monitoring, on-call). And every "stale answer for six hours" page starts with "the sync was lagging."
          ✓ Atlas: mongot tails the oplog. Index freshness is a property of the system.
        
        
        
        
          Seam 5 · embedding + rerank on the same surface
          First-party embedding and reranker models — including contextualized chunk embeddings (global document context captured at embedding time, not bolted on with metadata-prepending hacks) and dimension/quantization knobs — on the same ops surface, billed by the same vendor?
          If no: the embedding step itself is a separate vendor with its own latency budget, SOC2, billing — and you've baked in a context-agnostic embedding architecture that the next generation of contextualized models will eventually force you to migrate off of, across all your separate systems at once.
          ✓ Atlas: Voyage voyage-3 / voyage-3-large / voyage-context-3 embedders (Matryoshka 2048↔256, fp32↔int8↔binary) + rerank-2 rerankers, in-database. voyage-context-3 (binary, 512) beats OpenAI v3-large (float, 3072) at 0.5% of the storage cost.

        Two layers the engine-equivalence panel below doesn't cover
        
          
        
            The engine-equivalence panel below proves layers 1–3 are commodity — same math, same algorithm, same library class, ~95-99% top-K agreement across HNSW implementations. That's the premise of this whole demo, not the conclusion.
          
        
          
        
            What the engine panel can't show — because nobody's "engine" covers it — is the two layers at the ends of the stack:
          
        
          
        
            L0 — what you embed and which model embeds it. Contextualized embedding models like voyage-context-3 need to see the whole document at embedding time so each chunk's vector encodes both local content and global context. The line between "the data" and "the embedding model" has collapsed — the embedding step is now a data-modelling decision that wants to live next to the documents it embeds. Atlas ships Voyage as first-party for exactly this reason.
          
        
          
        
            L5 — the query processor. Pre-filter pushdown into HNSW, $lookup to sibling collections, server-side hybrid fusion via $rankFusion, server-side $rerank stages, $facet + $group rollups, projection — these are not drop-in between products. The composable-pipeline panel above is what L5 looks like in MongoDB; the "raw Lucene + glue" column is what L5 costs you when the query processor is your application code.
          
        
          
        
            The whole point: portability is strongest in the middle (L1–L3); differentiation lives at both ends (L0 and L5); and L4 — database integration — is what connects the two ends to the rest of your data. Every refinement to the stack diagram makes the agentic-database argument bigger, not smaller.

        An analogy. Every "specialized DB" wave was a different plug shape. The industry's answer was a new wall socket every time. That's why this table reads as one platform absorbing fifteen years of data-shape waves, instead of fifteen years of incompatible add-ons.
        "MongoDB isn't a vector database" is the headline. "MongoDB Atlas absorbs complexity into the platform and builds convenience into the query language" is the actual claim. The demo proves one slice of it. The tiles above are the rest.

            Hybrid retrieval · BM25 ⊕ vector ⇒ RRF · plus a real BooleanQuery
            
            Atlas exposes $rankFusion (rank-level). The score-level column is what Atlas doesn't.
          
          
            
            
            
              The same Lucene index that holds the HNSW vectors also runs BM25 over
              the plain-text field, so both retrievers read the same segments off the same
              mongot process. That's the bonus, and also the limit. When a
              user query mixes literal keywords and meaning, the demo runs both and fuses them
              with Reciprocal Rank Fusion: score = Σ wₚ · 1/(60 + rankₚ).
            
            

            
            
              Honest caveat. Atlas exposes
              $rankFusion for the rank-level version of this server-side
              (with per-pipeline weights), and $rerank for cross-encoder reranking on
              top — but it does not expose a score-level hybrid the way
              Solr's function_score / bf / bq or
              OpenSearch's hybrid query do. You can't ask mongot for
              0.6·textScore + 0.4·vectorScore in one Lucene query, and
              $search.compound won't take $vectorSearch as a clause.
              Two stages plus rank-level RRF is the contract; the
              BooleanQuery(KnnFloatVectorQuery, TermQuery) primitive Lucene itself
              supports is not surfaced. The third column below is exactly that — ~30 lines of
              Java against the same Lucene primitive that mongot runs, doing what

        appendix · the honest position
      
                  
      
                  
        When a specialized vector database is the right answer
      
                  
      
                  
        This whole demo argues vector databases are the wrong shape for AI agents. That argument is true.
        But it would be intellectually dishonest to pretend specialized vector databases — pure ANN services,
        large-scale vector tiers, search stacks with first-class kNN — were built by people who didn't know what they were doing. They ship in production today
        against workloads where they're absolutely the right pick. An honest decision tree is more useful to you than
        a triumphant headline, so here's both halves. (The page hasn't been trying to sell you a database; it's
        been trying to keep your AI infrastructure from collapsing into a pile of duct-taped microservices. The honest
        appendix is part of the same job.)
      
                  
      
                  
        Use a vector database when your product
        is vector search. Use an
        agentic database when your product
        is an agent.
      
                  
      
                  
        
                  
          
                  
            ✓ vector DB is right
          
                  
          
                    
            
                  • Recommendation systems · image-similarity · face search · plagiarism · fraud-by-similarity · chemistry · audio fingerprinting
                    • 
            
                  • Pure semantic-search features (no agent reasoning around them)
                    • 
            
                  • 1B+ vectors / 10K+ QPS with serious cost-per-query pressure
                    • 
            
                  • Brownfield: existing OLTP can't be migrated, vector recall needed now
                    • 
            
                  • 2-week prototype where speed-to-ship beats architectural purity
                    • 
          
                  
        
                  
        
                  
          
                  
            ✗ vector DB is wrong (use an agentic DB)
          
                  
          
                    
            
                  • AI agents — anything that reads operational records + chat + memory + traces on every turn
                    • 
            
                  • Greenfield AI app, no incumbent OLTP, choosing now
                    • 
            
                  • Anything where the "rescheduling your flight" walkthrough has 3+ steps that aren't step 3
                    • 
            
                  • Per-tenant scoped retrieval where filter pushdown into HNSW matters for correctness
                    • 
            
                  • Anything where "we'll just CDC the vectors over" is on a slide somewhere
                    • 
          
                  
        
                  
      
                  

      
      
                  
        
                  
          ⚠ where Atlas itself is still catching up to Solr / OpenSearch
        
                  
        
                  
          The "same Lucene index" line in the panel above is true. The sharper claim — "same Lucene query" — isn't yet. $rankFusion is rank-level, not score-level. Things the Lucene engine already supports that Atlas doesn't currently expose at the user-facing query layer:
        
                  
        
                    
          
                  • Score-level hybrid. No native linear combination of textScore and vectorScore with custom weights. $rankFusion fuses ranks (RRF with optional per-pipeline weights), not raw scores. Solr's function_score / bf / bq and OpenSearch's hybrid query both do score-level fusion at the engine.
                    • 
          
                  • One BooleanQuery across modalities. $search.compound doesn't accept $vectorSearch as a clause. You get two aggregation stages, not one Lucene BooleanQuery(KnnFloatVectorQuery, TermQuery) — even though that's exactly the primitive sitting one layer down. The third column in the panel above runs that primitive directly, in ~30 lines of Java.
                    • 
          
                  • Learning-to-rank. No first-party LTR training loop equivalent to Solr's LTR plugin or OpenSearch's ml-commons. $rerank calls a hosted Voyage cross-encoder — great for off-the-shelf reranking, not a model you fit on your own click data.
                    • 
          
                  • Field-level boost functions. Limited compared to Solr's function_score / bf/bq. You compose with $addFields + $sort in the pipeline, which is fine, but it isn't "in the Lucene query".
                    • 
        
                  
        
                  
          None of this changes the platform argument the rest of the page makes — the composable-pipeline panel is still the workload most teams actually have. But on the engine-equivalence panel specifically, "same Lucene index" is real and "same Lucene query" is not, and pretending otherwise wouldn't survive a serious search team's read.
        
                  
      
                  
      
                  
        The honest version, in one sentence:
        the "vector database" category is correct for the workload it was named for — pure semantic search where vector recall is the whole job — and a dedicated vector layer is exactly the right shape there. It's just no longer the most common AI workload — which is what makes "vector database" a misleading default for the agent-shaped systems most teams are actually shipping in 2026, and a misleading default for the platform-shaped data layer most non-agent systems are also outgrowing.
      
                  
      
                  
        And one more:
        same-Lucene-index doesn't yet mean same-Lucene-query. $rankFusion is rank-level fusion; score-level weighted hybrid (Solr function_score, OpenSearch hybrid) and a BooleanQuery-style cross-modality clause are parts of the engine Atlas hasn't exposed yet. The bonus is real; it isn't the whole story. The third column in the panel above is exactly the part Atlas still leaves on the floor.
